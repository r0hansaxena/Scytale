## Architecture Principles

- Each person should have their own Atsign (e.g. @alice, @bob) — this is the core identity unit.
- Agents and applications can either have a dedicated Atsign OR share an Atsign with their owner, using namespaces for isolation. Multiple agents/apps on one Atsign use different namespaces to keep their data separate.
- Multiple instances of the same agent can run under a single Atsign (see Multi-Agent Coordination below).
- IoT devices can have their own Atsign for autonomous operation, or be managed as keys under a person's Atsign.
- All data is end-to-end encrypted by default. Only the Atsign owner holds the decryption keys.
- There is NO application backend. atServers act as personal data stores and rendezvous points.
- Use namespaces to separate application data (e.g. "myapp" becomes key.myapp@atsign).
- Use AtKey with `sharedWith` to share encrypted data with specific Atsigns.
- Use AtKey with `metadata.isPublic = true` only for intentionally public data.
- Do NOT store application data in local files. Store data as AtKeys so it automatically synchronizes across all of the Atsign owner's devices. Local files are invisible to other devices; AtKeys flow via sync.

## Atsign Validation — All Dart Code (CLI, Agents, and Flutter)

- Always use at_commons to validate and parse every String used as an Atsign — this applies to ALL Dart code, including Flutter apps. For example, `'@bob'.toAtsign()` ensures the Atsign is valid before use.
- API parameters that expect Atsigns can accept plain strings, but always validate them too. For example: `atClient.getCurrentAtSign()?.toAtsign()`
- Use `.toAtsign()` precisely — the method is case-sensitive. Anything else will fail at runtime. This String extension is provided by at_commons, which is included with at_client and at_client_flutter.

## Initialization Pattern (Dart CLI / Agents)

- Use `AtClientManager.getInstance().setCurrentAtSign(atSign, namespace, preferences)` to initialize.
- Set AtClientPreference with `rootDomain: AtRootDomain.atsignDomain`, `hiveStoragePath`, `commitLogPath`, and `namespace`.
- IMPORTANT: When running multiple agents or agent instances on the same machine, each MUST use a unique hiveStoragePath and commitLogPath (e.g. a temp directory with a UUID: `Directory.systemTemp.createTempSync("agent_").path`). If two processes share the same hive path, the hive boxes will collide and throw errors.
- For agent/CLI authentication use at_cli_commons (CLIBase): `AtClient atClient = (await CLIBase.fromCommandLineArgs(args)).atClient;` — handles arg parsing, atKeys loading, and PKAM auth in one line.
- APKAM enrollment for new devices is supported via EnrollmentService in at_client.

## Flutter App Requirements

- Use at_client_flutter (NOT at_onboarding_flutter which is deprecated and unmaintained).
- Model your auth/onboarding screen on the at_client_flutter example app: https://github.com/atsign-foundation/at_client_sdk/tree/trunk/packages/at_client_flutter/example
- When using file_picker, add all required platform permissions: macOS entitlements (user-selected.read-only + read-write in both Debug and Release), iOS Info.plist usage descriptions, and Android READ_EXTERNAL_STORAGE/WRITE_EXTERNAL_STORAGE manifest permissions.
- For macOS, add com.apple.security.network.client and com.apple.security.network.server to both DebugProfile.entitlements and Release.entitlements; for iOS, add NSLocalNetworkUsageDescription to Info.plist.
- Study both main.dart (UI/navigation) and walkthrough.dart (all auth flows with error handling) in the example app.

### Welcome / Auth Screen (must support all four workflows)

1. **Login from Keychain**: `KeychainStorage().getAllAtsigns()` → `AtSignSelectionDialog(existingAtSigns: list)` → `AtAuthRequest(atSign, atKeysIo: KeychainAtKeysIo())` → `PkamDialog` → `setCurrentAtSign()`.
2. **Onboard a New Atsign (Registrar flow)**: Follow the `onboard()` function in walkthrough.dart from the at_client_flutter example app. Flow: `AtSignSelectionDialog.show(context)` → `RegistrarCramDialog.show(context, authRequest as AtOnboardingRequest, registrar: registrar)` → `CramDialog.show(context, request: authRequest, cramKey: cramKey)` → `_setupAtClient()`. Use `RegistrarService(registrarUrl: "my.atsign.com", apiKey: "5f93a2fa-2e3b-4332-9924-c29cc6e164ba")`.
3. **APKAM enrollment (new device)**: `AtSignSelectionDialog` → `ApkamActivationDialog.show(atSign, rootDomain, appName, deviceName, namespaces)` → waits for approval on an already-authorized device → `AtAuthRequest(atSign, atAuthKeys: enrollmentResponse.atAuthKeys)` → `PkamDialog` → `setCurrentAtSign()`.
4. **Login via .atKeys file**: `AtKeysFileDialog.show()` returns FileAtKeysIo → extract atSign from filename → `AtAuthRequest(atSign, atKeysIo: fileAtKeysIo)` → `PkamDialog(backupKeys: [KeychainAtKeysIo()])` → `setCurrentAtSign()`.

### Post-Auth Setup (all workflows end here)

- Get app support directory, create AtClientPreference with rootDomain, namespace, hiveStoragePath, commitLogPath.
- Call `AtClientManager.getInstance().setCurrentAtSign(atSign, namespace, prefs, atChops: response.atChops, atLookUp: response.atLookUp)`.

### Mandatory First-Run Atsign Gate (do not skip)

- Applications on this platform require Atsigns to function. Before showing ANY authentication UI, onboarding flow, login screen, or main application UI, implement a blocking First-Run Atsign Gate screen.
- The screen MUST appear automatically on first launch if `KeychainStorage().getAllAtsigns()` returns empty, and the user MUST NOT reach any auth workflow until they confirm.
- Content: title "Using this app requires an Atsign.", body explaining that existing users click "Continue" and new users get free, temporary Atsigns via the Starter Pack (https://my.atsign.com/starterpack_app — enter email, verify one-time passcode, return and click "Continue").
- Primary button "Get My Starter Pack" opens https://my.atsign.com/starterpack_app in the browser; "Continue" proceeds to the auth screen.
- Persist a local flag only after at least one Atsign exists on the device. The screen must feel intentional and part of the product.

## AtKey Construction

- Build keys with the AtKey builder pattern:
  - Self key (private, syncs across owner's devices): `AtKey()..key = 'name'..namespace = 'myapp'..sharedBy = me`
  - Shared key (E2E encrypted for one recipient): `AtKey()..key = 'name'..namespace = 'myapp'..sharedBy = me..sharedWith = '@bob'`
  - Public key (readable by anyone): `AtKey()..key = 'name'..namespace = 'myapp'..sharedBy = me..metadata = (Metadata()..isPublic = true)`
- Useful metadata: `ttl` (expiry ms), `ttb` (available-from ms), `ttr` (recipient cache refresh), `immutable = true` (write-once, used for mutexes), `isEncrypted`.
- Full key names look like `@bob:msg.abc123.myapp@alice` (shared), `msg.abc123.myapp@alice` (self), `public:profile.myapp@alice` (public).

## Data Operations

- `atClient.put(key, value)` creates/updates an AtKey, stores it locally, and auto-syncs to the atServer when online.
- To write directly to the cloud secondary, pass `putRequestOptions: PutRequestOptions()..useRemoteAtServer = true`.
- `atClient.get(key)` reads a value; pass `getRequestOptions: GetRequestOptions()..useRemoteAtServer = true` to read from the remote secondary, or `..bypassCache = true` to skip cached values.
- `atClient.getAtKeys(regex: ..., sharedBy: ..., useRemoteAtServer: ...)` scans for keys; with `sharedBy` set to another Atsign it lists keys that Atsign shared with you.
- `atClient.delete(key)` removes a key.
- NEVER store application data in local files — AtKeys sync across devices, local files do not.

## Communication Patterns

### 1. Async Communication (fire-and-forget)

- Use `atClient.put()` to store data for another Atsign without requiring them to be online — the data is written to your atServer and synced to theirs when they next connect.
- The receiving Atsign retrieves it later with `atClient.get()`.
- This is fire-and-forget: ideal when you do not know if the recipient is online.
- See: https://pub.dev/documentation/at_client/latest/at_client/AtClient/put.html and https://pub.dev/documentation/at_client/latest/at_client/GetRequestOptions-class.html

### 2. Sync Communication (delivery confirmation)

- Use `notificationService.notify()` with `NotificationParams.forUpdate(key, value: data)` to send data and wait for a delivery confirmation.
- The sender gets a `NotificationResult` indicating delivered/errored status.
- Best when you need confirmation that the remote Atsign received the data.
- Combine with `notificationService.subscribe()` on the receiving side.
- See: https://pub.dev/documentation/at_client/latest/at_client/NotificationService/notify.html

### 3. Notification / Event-driven Communication

- For near real-time event-driven communication between Atsigns.
- Receiver subscribes with: `atClient.notificationService.subscribe(regex: "pattern", shouldDecrypt: true).listen((notification) { ... })`.
- Sender pushes with: `atClient.notificationService.notify(NotificationParams.forUpdate(key, value: data))`.
- Notifications are lightweight, support regex filtering, and auto-reconnect on network loss.
- See: https://pub.dev/documentation/at_client/latest/at_client/NotificationService/subscribe.html

### 4. RPC (Remote Procedure Call)

- Use the built-in `AtRpc` and `AtRpcClient` classes for structured request-response patterns.
- Create an `AtRpc` with `AtRpcCallbacks` (handleRequest, handleResponse) and an allowList of permitted Atsigns.
- The requester calls `rpc.sendRequest(toAtSign: target, request: AtRpcReq(...))` and receives responses via the `handleResponse` callback.
- Uses Atsign Protocol notifications under the hood.
- For multi-instance agents: use an immutable mutex race (`Metadata()..immutable = true`, put with `useRemoteAtServer = true`) so only one instance handles each request — `AtRpc.server(..., enableRequestMutex: true)` has this built in. Alternatively, use `ServiceFactoryWithNoOpSyncService()` with per-session ephemeral storage for stateless horizontally-scaled instances.
- See: https://pub.dev/documentation/at_client/latest/at_client/AtRpc-class.html and https://pub.dev/documentation/at_client/latest/at_client/AtRpcClient-class.html

### 5. Streaming Data

- For continuous data flows use `notificationService.subscribe(regex: "stream.*", shouldDecrypt: true)` for receiving updates, and `notificationService.notify()` in a loop or timer on the sender side.
- For bulk file transfer: encrypt the file client-side, upload to a web file service, then send the URL and encryption key as an encrypted AtKey — the recipient downloads and decrypts locally.
- Note: `atClient.startMonitor()` is deprecated — always use `notificationService.subscribe()` instead.
- For streaming LLM answers the new https://pub.dev/packages/at_stream can be used although it is pre-release.

### 6. TCP / Socket Communication (NoPorts)

- For E2E encrypted TCP connections with zero open listening ports, use NoPorts (noports_core package).
- It creates a local tunnel port that routes traffic through the Atsign Platform to a remote device — no firewall changes needed on either side.
- CLI: `npt` binary. Programmatic: `Npt` class.
- See: https://pub.dev/packages/noports_core and https://docs.noports.com/

## Multi-Agent Coordination

When running multiple instances of the same agent on the same Atsign, use ONE of these patterns:

1. **Mutex race with immutable keys**: First instance to write an immutable key (`Metadata()..immutable = true`) with `useRemoteAtServer = true` wins — that instance handles the task. Other instances see the key exists (a `put()` returning false / throwing) and skip the task. Use TTL to auto-release the lock.
2. **ServiceFactoryWithNoOpSyncService**: Create agent instances with ephemeral per-session storage (no sync). For stateless processing like request-response agents that do not need to persist state across restarts.

Also: every instance on the same machine MUST use a unique hiveStoragePath/commitLogPath (see Initialization Pattern).

## Networking, File Transfer & Access Control

### TCP Tunnels

- NoPorts (noports_core) provides E2E encrypted TCP tunnels through the Atsign Platform with zero open listening ports on either side — no firewall changes, no port forwarding.
- Use the `npt` CLI or the programmatic `Npt` class to open a local tunnel port that routes to a remote device or service.
- See: https://docs.noports.com/

### Encrypted File Transfer

- For bulk/large files, do NOT push the bytes through notifications. Encrypt the file client-side, upload the ciphertext to any web file service, then share the URL + decryption key with the recipient as an encrypted AtKey.
- The recipient downloads the ciphertext and decrypts locally. The file service never sees plaintext.
- Small payloads can be embedded directly in AtKey values (base64), subject to atServer value-size limits.

### Policy Enforcement

- APKAM enrollment scopes what each app/device may access: enrollments request specific namespaces with read/write permissions (e.g. `{'myapp': 'rw'}`), approved from an already-authorized device.
- Use `EnrollmentService` (at_client) to list, approve, and revoke enrollments — revoking a device's enrollment cuts off its access without affecting other devices.
- AtRpc `allowList` restricts which Atsigns may invoke an RPC endpoint; requests from other Atsigns are discarded before reaching the handler.
- Namespace isolation is the primary access-control boundary between apps/agents sharing an Atsign.

## Key Conventions

- Namespace every key: `key.namespace@atsign`. One namespace per app or agent.
- Use short dotted prefixes inside the namespace to type keys (e.g. `msg.<id>`, `profile`, `lock.<id>`).
- Prefer JSON-encoded values with explicit schemas documented in the project README.
- Use `Metadata()..ttl` for ephemeral data (locks, presence) and `immutable = true` for write-once records/mutexes.
- Validate every Atsign string with `.toAtsign()` before using it in a key.

## Error Handling

- Always wrap AtClient operations in try-catch.
- Common exceptions: AtClientException, AtKeyException, AtConnectException.
- Check `AtClientManager.getInstance().atClient.syncService.syncInProgress` before critical operations.
